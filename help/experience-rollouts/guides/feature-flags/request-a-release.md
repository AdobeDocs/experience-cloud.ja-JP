# リリースをリクエスト {#request-a-release}

## 前提条件 {#prerequisites}

* **リリースマネージャー**&#x200B;の役割があります – [ ユーザーロール ](../teams/user-roles.md)を参照してください
* アプリケーションがオンボーディングされました。[ アプリケーションのオンボーディング ](../applications/onboard-your-application.md)を参照してください。

>[!TIP]
>
>リリースをリクエストする前に、[ リリースとチーム間の機能グループ ](releases-and-cross-team-feature-groups.md)を確認してください。 クロスチームの機能グループは、セルフサービス型で、より豊富なオーディエンスターゲティングをサポートするため、より少ないオーバーヘッドでニーズを満たすことができます。

## サポートリクエストを送信 {#submit}

リリース作成はセルフサービスではありません。 Experience Rollouts サポートに連絡して、新しいリリースをリクエストしてください。 次の情報を入力します。

| フィールド | 説明 |
|---|---|
| **名前** | リリースの一意の名前。 リリース名にスペースを含めないでください。 |
| **チーム** | リリースのオーナーとなるチーム。 |
| **環境** | リリースを作成する環境（ステージングまたは実稼動環境）。 |
| **目標** | このリリースの内容とその理由を簡単に説明します。 |
| **アプリケーション** | リリースに含めるアプリケーション。 |
| **期間** | リリースをアクティブな状態に保つのに必要な時間。 リリースは、3か月以内にクローズまたはベースライン化される予定です。 必要な場合は、ビジネス上の理由を説明してください。 |

>[!NOTE]
>
>任意の時点でのアクティブなリリースの数は制限されています。 他のチームのキャパシティを増やすために、3か月以内にリリースをベースライン化またはクローズする予定です。

## リリースの作成後 {#after}

リリースが作成されたことを確認したら、コンソールにログインしてリリース設定を完了します。 一連の手順について詳しくは、[ リリースワークフローのエンドツーエンド ](release-workflow-end-to-end.md)を参照してください。
