# リリースとチーム間の機能グループ {#releases-and-cross-team}

リリースとチーム間の機能グループはどちらも、複数のチームやアプリケーションをまたいで調整されたロールアウトをサポートします。 適切な選択とは、キャッシュの要件、オーディエンスのターゲティングのニーズ、プロセスをどの程度コントロールするかによって異なります。

## チーム間の機能グループ {#cross-team-feature-groups}

チーム間の機能グループを利用すれば、異なるチームが所有するアプリケーションから機能フラグをバンドルし、豊富なオーディエンス基準にもとづいて管理できます。

主な特徴：

* **セルフサービス** – 作成するサポートリクエストは必要ありません
* **リッチオーディエンスターゲティング** — オーディエンス条件の完全なセット（プロファイル属性、コンテキスト変数、パーセンテージのロールアウト）をサポートします
* 作成できる数に&#x200B;**制限**&#x200B;はありません
* **キャッシュなし** – 機能フラグの評価にはAPI呼び出しが必要です。SDK レベルのキャッシュはサポートされていません

ステップバイステップの作成手順については、[&#x200B; チーム間の機能グループの作成](create-cross-team-feature-group.md)を参照してください。

## リリース {#releases}

リリースは、SDK レベルでのキャッシュが必要な、大規模で調整されたローンチ用に設計されています。 リリースでは、認証トークンベースのオーディエンスターゲティングを使用します。

主な特徴：

* **サポートリクエストが必要です** — リリースは完全にセルフサービスではありません。Experience Rollouts サポートに連絡して作成してください
* **SDK キャッシュ** – すべてのリリースデータはサーバーSDKにローカルにキャッシュされ、API呼び出しが減少します
* **オーディエンス条件の制限** – 認証トークン（国、電子メール、ユーザーID、パーセンテージなど）に基づいてオーディエンスルールを設定します。
* **制限付きアクティブ リリース** — 1回に限られた数のアクティブ リリースが存在する可能性があります。3か月以内にリリースをクローズまたはベースライン化する予定です

## 比較 {#comparison}

| | チーム間の機能グループ | リリース |
|---|---|---|
| セルフサービス | ✓ | サポートリクエストが必要 |
| 豊富なオーディエンス基準 | ✓ | 制限付き |
| A/B テスト | ✗ | ✗ |
| SDKのキャッシュ | ✗ | ✓ |
| アクティブ制限 | 制限なし | 制限付き |

## レコメンデーション {#recommendation}

ユースケースでリッチなオーディエンスのターゲティングとセルフサービス管理が必要な場合は、**チーム横断機能グループ**&#x200B;を使用します。 バックエンドサービスでSDK レベルのキャッシュが必要で、リリースのシンプルなオーディエンス条件で十分な場合は、**リリース**&#x200B;を使用します。
